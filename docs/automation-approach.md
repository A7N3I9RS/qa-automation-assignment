# Automation Approach

## Architecture

The project uses Playwright Test with TypeScript.

SauceDemo UI tests use typed Playwright fixtures from [src/saucedemo/fixtures](../src/saucedemo/fixtures) to provide page objects from [src/saucedemo/pages](../src/saucedemo/pages). This keeps selectors and page-level actions away from test assertions without recreating page objects inside every test. Shared SauceDemo users and customer data live in [src/saucedemo/data](../src/saucedemo/data).

ReqRes API tests use Playwright's built-in `request` fixture and external test data from [tests/reqres/data](../tests/reqres/data).

## Why This Coverage Was Prioritized

The first automation priority is the main business flow:

```text
User buys a product
```

That flow gives the highest regression value because it crosses authentication, product selection, cart review, checkout information and order completion.

After the happy path, the suite adds targeted risk coverage:

- Locked-out login behavior
- Catalog sorting
- Cart state changes
- SauceDemo special-user differences
- API data and schema checks
- API create-user response-time check

## Handling Known Defects

SauceDemo intentionally exposes defects through dedicated users such as `problem_user`, `performance_glitch_user`, `error_user` and `visual_user`.

Cart and checkout specs run their functional scenarios against users with access. Checkout visual tests also run for each user with access, comparing them against the `standard_user` baseline so user-specific visual differences appear in the report.

## Test Commands

| Command                                       | Purpose                                             |
| --------------------------------------------- | --------------------------------------------------- |
| `npm test`                                    | Run every Playwright project                        |
| `npm run test:saucedemo:all-browsers`         | Run SauceDemo tests in Chromium, Firefox and WebKit |
| `npm run test:saucedemo:chromium`             | Run SauceDemo tests in Chromium                     |
| `npm run test:saucedemo:chromium:headed`      | Run SauceDemo Chromium tests headed                 |
| `npm run test:saucedemo:firefox`              | Run SauceDemo tests in Firefox                      |
| `npm run test:saucedemo:webkit`               | Run SauceDemo tests in WebKit                       |
| `npm run test:saucedemo:update-snapshots`     | Update standard-user checkout visual snapshots in all UI browsers |
| `npm run test:reqres`                         | Run all ReqRes tests                                |
| `npm run test:saucedemo:baseline`             | Run SauceDemo baseline coverage                     |
| `npm run test:ui-mode`                        | Open Playwright UI mode for local test development  |
| `npm run show-report`                         | Open the Playwright HTML report                     |

## Reporting and Debugging

Playwright is configured to retain useful artifacts on failure:

- Trace: retained on failure
- Screenshot: only on failure
- Video: retained on failure
- HTML report: generated by Playwright
- GitHub Pages: publishes the latest `main` branch HTML report for easier review

Visual snapshots use the project name in their path, so Chromium, Firefox and WebKit keep separate baselines and never compare screenshots across browsers. The checkout visual baseline is captured with `standard_user`; other users with access compare against that same browser-specific baseline.

Slow motion is opt-in through `PLAYWRIGHT_SLOW_MO_FLAG=1`. `PLAYWRIGHT_SLOW_MO_MS_TIME` only controls the delay amount and defaults to `1000` milliseconds.

## CI

The repository includes a GitHub Actions workflow at [.github/workflows/playwright.yml](../.github/workflows/playwright.yml).

CI runs `npm test`, the full Playwright suite. This keeps defects visible in pull requests and preserves the HTML report as triage evidence. Pull requests keep the report as a workflow artifact; `main` branch runs also deploy the report to GitHub Pages.

To run the suite in CI, configure repository secrets named `USERS_PASSWORD` and `REQRES_API_KEY`.
